# Travel Intelligence OS - Execution Engine Specification

This document defines the architecture, pipeline flow, failure handling, and state machine of the master Execution Engine (`backend/execution/execution_engine.js`). It coordinates all engines (Compiler, Planner, Decision, Optimizer, Budget, Recommendation, Booking, and Conversation) in a deterministic execution sequence.

---

## 1. Purpose

The Execution Engine serves as the single entry point and master orchestrator for processing every travel request in the system. It manages the lifecycle of context execution, enforces conditional gating (such as clarification locks or no-op skipping), and ensures that execution terminates safely with a unified response contract.

---

## 2. Responsibilities

- **Pipeline Coordination**: Execute active stages in the correct logical order.
- **Conditional Gating**: Skip engines when updates are not relevant (e.g., skip `RouteOptimizer` if transport has not changed).
- **Execution Lifecycle Control**: Freeze execution when `ClarificationEngine` demands user inputs.
- **State Merging**: Safely merge engine outputs into separate namespace domains inside the `TravelContext` without collision.
- **Failure Gating**: Handle recoverable vs. critical pipeline errors and execute retries/rollbacks.
- **Unified Contract Output**: Guarantee the final payload returns the standard Engine Response Contract.

---

## 3. Inputs

- **`TravelContext`**: The core state and request container.
- **`PipelineManager Output`**: Compiled parameters from incoming text queries.
- **`WorkflowManager Output`**: Orchestration rules for the conversation.
- **`Conversation State`**: Tracks current conversation phase.
- **`Execution State`**: Tracks executed/skipped engines in the active thread.
- **`Planner Instructions`**: Directives generated by the `ReplanningEngine`.

---


## 5. Engine Registry

The Execution Engine loads and calls engines dynamically via a deterministic registry. No hardcoded switch statements are allowed.

### Registered Modules
- `classifier`: `backend/conversation/conversation_classifier.js`
- `state`: `backend/conversation/conversation_state.js`
- `updater`: `backend/conversation/context_updater.js`
- `clarification`: `backend/conversation/clarification_engine.js`
- `replan`: `backend/planner/replanning_engine.js`
- `planner`: `backend/planner/trip_planner.js`
- `decision`: `backend/decision/decision_engine.js`
- `optimizer`: `backend/optimizer/route_optimizer.js`
- `budget`: `backend/budget/budget_engine.js`
- `recommendation`: `backend/recommendation/recommendation_engine.js`
- `booking`: `backend/booking/booking_engine.js`

## 6. Execution Order

The engine executes in a strict deterministic order:

```
[User Input Query]
        │
        ▼
1. Conversation Classifier (Identify Intent & Type)
        │
        ▼
2. Context Updater (Mutate Entities)
        │
        ▼
3. Clarification Engine (Core Info Verification Gate)
        │
        ├───► WAITING_CLARIFICATION ──► [Prompt Clarification Prompt & Block]
        │
        ▼
4. Replanning Engine (Change Scope & Downstream Analysis)
        │
        ▼
5. Trip Planner (Itinerary Base Day Generation)
        │
        ▼
6. Decision Engine (Heuristics Optimization Swapping)
        │
        ▼
7. Route Optimizer (TSP Solver & Transport Selection)
        │
        ▼
8. Budget Engine (Cost Breakdown & Cost-Saving suggestions)
        │
        ▼
9. Recommendation Engine (Enrichment - Packing, Safety, Customs)
        │
        ▼
10. Booking Intelligence (Booking options & Provider score ranking)
        │
        ▼
[Final Unified Response output]
```

---

## 7. Conditional Execution

To optimize computing resources, the Execution Engine reads directives and skips stages:
- **Clarification stops**: If `ClarificationEngine` returns `requiresClarification: true`, sets state to `WAITING_CLARIFICATION`, stops execution immediately, and skips downstream stages.
- **Budget-only changes**: If `ReplanningEngine` outputs only `["REPLAN_BUDGET"]`, runs only `BudgetEngine` and updates context.
- **No-op modifications**: If `requiresReplanning` is `false`, bypasses planner, decision, and optimizer, directly loading recommendations from history cache.
- **No bookings requested**: If the intent is planning only, skips the `BookingEngine` stage.

---

## 8. Failure Strategy

- **`Recoverable Failures`**: If recommendation or booking fails, logs warning, skips the stage, and sets state to `COMPLETED` with partial success warnings.
- **`Critical Failures`**: If planner, optimizer, or state updater fails, immediately halts, sets state to `FAILED`, and rolls back context changes.
- **`Rollback`**: Restores `context.state` to previous values if critical failures occur mid-pipeline.

---

## 9. Context Merging

Each engine owns its namespace under `TravelContext`:
- `context.state.normalizedEntities`: Controlled by `ContextUpdater`.
- `context.state.conversationState`: Controlled by `ConversationStateManager`.
- `context.recommendations.draftItinerary`: Created by `TripPlanner`.
- `context.recommendations.improvedItinerary`: Created by `DecisionEngine`.
- `context.recommendations.optimizedItinerary`: Created by `RouteOptimizer`.
- `context.recommendations.budgetSummary`: Created by `BudgetEngine`.
- `context.recommendations.bookingSuggestions`: Created by `BookingEngine`.

---

## 10. Execution State Machine

```
      Start Call
[READY] ───────────────► [RUNNING]
                           │
                           ├───► Clarification Required ──► [WAITING_CLARIFICATION]
                           │
                           ├───► Critical Error ──────────► [FAILED]
                           │
                           ├───► Cancel/Reset ────────────► [CANCELLED]
                           │
                           └───► Execution Finished ──────► [COMPLETED]
```

---

## 11. Retry Strategy

- **Retries**: A maximum of 2 retries is allowed on recoverable engine failures (e.g. transient pricing provider errors inside `BookingEngine`).
- **Critical timeouts**: Individual stages must complete within 2000ms. If exceeded, halts execution and flags critical failure.

---

## 12. Parallel Execution

While execution from `Classifier` to `Optimizer` must remain synchronous due to day-dependency chains, the post-routing stages may run in parallel:
- `BudgetEngine`, `RecommendationEngine`, and `BookingEngine` do not depend on each other's parameters. Future architectures can invoke them using `Promise.all()` to improve processing speeds.

---

## 13. Logging

Log outputs must follow this standardized schema:

```json
{
  "stage": "route_optimizer",
  "startTime": "2026-07-14T01:50:00.000Z",
  "endTime": "2026-07-14T01:50:00.002Z",
  "durationMs": 2,
  "status": "COMPLETED",
  "warnings": [],
  "errors": []
}
```

---

## 14. Error Handling

- **Unknown Engine**: Fails pipeline immediately; registers critical failure.
- **Timeout**: Logs `Stage timed out` and aborts.
- **Cancelled Execution**: Instantly clears memory and state variables.

---

## 15. Future Extensibility

- **Distributed Workers**: Integrating message queues (e.g., BullMQ) to offload routes and booking checks to background jobs.
- **Multi-modal response composition**: Translating final compiled logs to visual Markdown/React maps.

---

## 16. Verification Plan

Verify the following:
1. **Complete Execution**: Successful planning journey from classifier to recommendations.
2. **Clarification stop**: Verify pipeline freezes on missing destination.
3. **Replanning path**: Verify only budget and recommendation runs on budget update.
4. **Planner failure**: Verify context rolls back to prior state on planner crash.
5. **Timeout Handling**: Force simulated delays to verify timeout abort.
6. **Parallel-ready checks**: Verify no cross-dependencies between budget and recommendations.
